# Raging Bolt LLM Strong Mode Architecture Upgrade Plan

Date: 2026-05-04
Owner: Codex
Target deck: Raging Bolt LLM strong mode
Validation opponent: rules Miraidon

## 1. Goal

Use Raging Bolt LLM strong mode as the reference implementation for the next LLM deck architecture. The goal is not to make a prompt longer. The goal is to make the whole path stable:

1. The LLM sees the current board, hand, legal action space, future tactical routes, and deck strategy.
2. The LLM chooses a route-style decision tree using only legal ids or candidate route ids.
3. The runtime materializes the chosen route into exact executable actions.
4. Interactions such as discard/search/attach/target choices are bridged deterministically.
5. Strong mode cannot let MCTS, learned scorers, or rules fallback silently override an active LLM queue.
6. Every completed match can be audited from prompt payload to response, queue selection, runtime execution, fallback, and final result.

Win rate is a final acceptance metric, not the first debugging signal. A run only counts as LLM evidence when request success and queue takeover are high.

## 2. Current Problem Statement

Recent Raging Bolt LLM matches show three recurring failure classes.

### 2.1 Low-Value Terminal Actions

Raging Bolt sometimes chooses the first attack, draw/filter actions, or end turn even when a visible route can set up the primary burst attack. This happens when the legal action exists but the route/fact layer does not clearly explain why setup before attack is better.

### 2.2 One-Shot Planning Under State Changes

The LLM may choose a valid initial line, but after draw/search/discard the hand and board change. If replan payloads do not include updated turn flags, current hand, current board, route prefix, and remaining queue, the second decision cannot repair the route correctly.

### 2.3 Execution Contract Gaps

Even when the LLM intent is reasonable, the executor can lose it if:

- The LLM uses invented ids or unsupported interaction keys.
- A future action is selected before it becomes executable.
- A route id is not materialized into exact action ids.
- A card-specific interaction bridge does not map the strategy into concrete legal choices.
- Strong-mode runtime falls through to another decision path while an LLM queue is active.

## 3. Architecture Layers

### 3.1 Prompt Payload Layer

File: `scripts/ai/LLMTurnPlanPromptBuilder.gd`

Responsibilities:

- Build compact but complete `game_state`.
- Include own hand and public board card rules from JSON.
- Include opponent board, HP, tools, energy, statuses, prizes, deck count, discard summary, and hand count.
- Include turn flags: manual energy attached, supporter used, retreat used, stadium used.
- Include `legal_actions` with exact ids, resource consumption, conflicts, card rules, attack rules, ability rules, and interaction schemas.
- Include `legal_action_groups` as a checklist: attack, attach, engine/draw, search/setup, pivot/gust, tool/modifier, fallback.
- Include `future_actions` only as same-turn goals after setup, not as immediately executable first steps.
- Include `turn_tactical_facts` that are generated by code, not guessed by the LLM.
- Include `candidate_routes` as the primary planning surface.
- Include `replan_context` after meaningful draw/search/discard/bench changes.

Design rule: deck strategy prompt gives tactical preference; structured facts and legal ids define what is currently true.

### 3.2 Tactical Fact Layer

File: `scripts/ai/LLMTurnPlanPromptBuilder.gd`

Responsibilities:

- Mark primary attack readiness and missing cost.
- Distinguish low-value redraw/setup attacks from primary damage attacks.
- Expose best manual attach for current and future attackers.
- Expose safe pre-primary actions such as Ogerpon ability when they add board energy and draw before attack.
- Expose `no_deck_draw_lock` and `deck_draw_risk`.
- Expose gust KO opportunities and game-ending prize routes.
- Expose resource-negative actions such as dead Lost Vacuum or dead gust.
- Expose Sada-style assignment recommendations that match missing attack costs.

Design rule: facts should be conservative and exact. If a human-visible line is not represented here, the LLM is unlikely to discover it reliably from raw JSON.

### 3.3 Candidate Route Layer

Files:

- `scripts/ai/LLMRouteCandidateBuilder.gd`
- `scripts/ai/LLMRouteCompiler.gd`
- `scripts/ai/LLMRouteActionRegistry.gd`

Responsibilities:

- Convert facts and legal actions into high-value `route:*` ids.
- Prioritize routes over shallow single-action lines.
- Materialize route ids back to exact legal action refs.
- Insert safe setup before terminal actions.
- Suppress low-value attack-now routes when productive setup, primary attack reachability, or deck draw risk exists.
- Avoid inserting draw/churn engines when deck count is risky.
- Keep search/setup routes available even under low-deck lock when they do not draw cards directly.
- Provide pivot-to-primary-attack routes for bench attackers that can attack after attach or retreat.

Design rule: LLM should usually select one route id. It should not duplicate route actions manually unless no route fits.

### 3.4 Runtime Ownership Layer

File: `scripts/ai/DeckStrategyLLMRuntimeBase.gd`

Responsibilities:

- Own ZenMux request lifecycle.
- Validate JSON and decision-tree contract.
- Materialize route ids and aliases.
- Apply safe repairs only when the correct stronger route is unambiguous.
- Compile a queue and score only that queue while active.
- Trigger replan after meaningful hand/board changes.
- Disable only the current turn on invalid JSON, timeout, or bad contract.
- Fall back to rules only when no valid current LLM plan exists.
- Audit request, response, repair, selected queue, runtime action, queue consumption, failure, and fallback.

Design rule: if `plan_selected.action_queue` is good but the game executes another first action, this is an ownership bug, not an LLM reasoning bug.

### 3.5 Interaction Bridge Layer

Files:

- `scripts/ai/LLMInteractionIntentBridge.gd`
- target deck LLM wrapper
- target deck rules strategy fallback

Responsibilities:

- Convert `selection_policy` and exact `interactions` into legal discard/search/attach/target choices.
- Reject unsupported generic keys when an action has no schema.
- Preserve route-critical cards during discard.
- Select energy types according to the target Pokemon's attack requirements.
- Let rules fallback handle revealed/random option pools the LLM could not know at planning time.

Design rule: card-specific execution is still deterministic. The LLM can choose strategy, but the bridge must make that strategy executable against the actual legal option pool.

## 4. Raging Bolt Template Semantics

Raging Bolt LLM should establish these deck-level semantics as a template:

- Raging Bolt ex is the main burst attacker.
- Teal Mask Ogerpon ex is safe setup when Grass is available and deck draw risk is not active.
- Radiant Greninja and Trekking Shoes are draw/filter engines, but they must not discard missing attack energy when a direct attach route is already available.
- Professor Sada's Vitality should attach Lightning/Fighting or needed basic energy to Ancient attackers according to attack cost gaps.
- Earthen Vessel should find missing Lightning/Fighting first, then Grass for Ogerpon when the attack cost is already covered.
- The first Raging Bolt attack is a low-value redraw/setup attack and should be suppressed when primary damage is ready or reachable.
- Boss/Catcher/gust should be chosen when it creates a KO or game-winning prize route. If it is merely legal but resource-negative, the fact layer should mark it and the prompt should deprioritize it. Hard runtime bans are not currently accepted unless benchmark-proven.
- When deck count is low, draw/churn is locked out, but deterministic search/setup that does not directly draw should remain available.

## 5. Implemented In This Upgrade

This document tracks the current upgrade round.

### 5.1 Low-Deck Draw Risk

Implemented:

- `no_deck_draw_lock` and `deck_draw_risk` use a practical threshold of 1 to 12 deck cards.
- Low-deck lock blocks optional draw/churn insertions.
- Low-deck lock does not globally block deterministic search or setup routes.

Reason:

The previous model could avoid immediate deckout too late. At low deck count, actions like Raging Bolt first attack, Greninja, Shoes, Sada, Iono, and Fezandipiti can become dangerous unless they are required for a winning route.

### 5.2 Low-Value Attack Route Suppression

Implemented:

- Candidate builder suppresses low-value attack-now routes when:
  - redraw attack is forbidden,
  - deck draw risk is active,
  - primary attack is reachable,
  - manual attach enables a better attack,
  - productive non-terminal setup exists.

Reason:

The LLM should not be tempted by a legal low-value attack when the current turn can build a stronger route.

### 5.3 Pivot-To-Primary-Attack Route

Implemented:

- Candidate builder can expose a route that charges or pivots to a bench attacker and lets runtime convert terminal end-turn into the now-legal attack.

Reason:

Many Raging Bolt failures come from active non-attacker or active Pokemon that cannot currently attack, while a bench Raging Bolt can become the attacker after attach/pivot.

### 5.4 Priority Candidate Route Repair

Implemented:

- Runtime can replace a weak selected tree with a stronger high-priority candidate route when:
  - the route priority is high,
  - the selected tree does not already contain that route,
  - the selected tree does not contain an immediate high-pressure attack.

Reason:

This is a safety rail for route ids. It should repair obvious route selection failures without turning the runtime into a second hand-written strategy engine.

### 5.5 Localized Card Matching

Implemented:

- Queue/catalog matching searches card id, localized name, English name, rule text, and effect id.

Reason:

Card names in UI, JSON, and LLM output can mix Chinese and English. Runtime matching must not fail just because `card_rules.name_en` is the only stable name.

### 5.6 Resource-Negative Gust Facts

Implemented:

- Catcher/Boss/Prime Catcher style gust actions are marked as `resource_negative_actions` when the current legal attacks contain no non-low-value damage attack.
- This is prompt/fact guidance, not a hard runtime ban.

Reason:

Hard-blocking gust at runtime reduced benchmark results. The correct architecture is to expose the current tactical truth to the LLM and reserve hard runtime blocks for cases that are proven by benchmark and have precise exceptions.

### 5.7 Specific-Cost Manual Attach Gate

Implemented:

- Candidate route generation prefers manual attach actions that reduce the active primary attack's missing cost.
- When the active primary attack still needs specific non-colorless energy, an off-cost active attach is not exposed as `manual_attach_setup`.
- Generic fallback attach is still allowed when no specific primary attack cost is known.

Reason:

Raging Bolt can convert any basic energy into burst damage, but it still must first satisfy the `Lightning + Fighting` attack cost. Exposing a Grass-to-active attach as the best setup route when Lightning/Fighting is still missing misleads the LLM into a non-attacking end turn. The right abstraction is not "any energy is useful"; it is "attack-cost energy first, board-damage energy only after attack access is secured."

### 5.8 Low-Deck Draw/Churn Hard Lock

Implemented:

- Candidate route generation removes draw/churn actions once `no_deck_draw_lock` is active.
- The route compiler also removes explicit LLM-selected draw/churn actions under the same lock.
- The lock recognizes normal tagged draw actions and schema-only abilities such as hand-energy charge plus draw, discard-to-draw, and filter draw.
- Raging Bolt LLM strong mode blocks unplanned rules-fallback Ogerpon/Greninja/Fezandipiti and major draw-supporter/filter actions when the deck is locked.

Reason:

Prompt-only warnings were not enough. In the 202605052 replay, the LLM continued using Ogerpon, Greninja, and filter cards after the deck fell to single digits, producing a deckout loss despite full LLM takeover. Low-deck draw/churn is a deterministic execution safety problem, so it belongs in the route and compiler layers.

The third bypass was rules fallback: when the selected LLM queue no longer matched, the inherited rules scorer could still choose a draw ability. Strong-mode runtime must guard that fallback path too.

### 5.9 Future Attack Goal Ownership And Blocked End-Turn Enforcement

Implemented:

- Runtime `end_turn` conversion now respects route compiler future attack goals.
- If a route goal says the terminal attack should come from Raging Bolt, `end_turn` cannot be converted into a support Pokemon's current attack.
- If the route compiler or deck hook says end turn is blocked, the runtime assigns a hard negative score to `end_turn` while an LLM queue is active.
- Raging Bolt LLM marks support Pokemon attacks such as Squawkabilly ex, Radiant Greninja, and Iron Bundle as low-value runtime attacks.
- Raging Bolt rules fallback now prefers sending out a backup Raging Bolt over a charged support Pokemon when attack continuity is available.

Reason:

The previous guard only made blocked `end_turn` fail queue matching. That was not enough: once queue matching failed, the inherited rules scorer could still choose `end_turn`, causing queue escape, repeated replan, and sometimes a stuck headless match. Blocked terminals must be actively negative-scored at the runtime ownership layer.

Future attack goals need source ownership because route compilation removes future attack refs from the executable queue. Without source ownership, a generic `end_turn` terminal could be interpreted as "use any high-pressure attack now", including a support Pokemon attack, even when the planned route was to pivot into Raging Bolt and use the primary burst attack.

### 5.10 Low-Deck Fact Consistency

Implemented:

- `no_deck_draw_lock` now filters prompt facts, not only route compilation and runtime fallback.
- `safe_pre_primary_actions` no longer advertises charge-and-draw abilities when deck draw is locked.
- `productive_engine_actions` no longer advertises Ogerpon, Greninja, Trekking Shoes, Fezandipiti, Sada/Iono-style draw lines under low-deck lock.
- Non-draw resource recovery, such as Night Stretcher, remains visible.

Reason:

The previous implementation was internally inconsistent. The prompt told the LLM that Ogerpon, Greninja, and Shoes were productive, then the route compiler removed those actions because the deck was low. That produced shallow routes like "attach Grass, end turn" and unnecessary replans. Facts, routes, compiler, and runtime must expose the same tactical truth.

### 5.11 Pending Interaction Ownership

Implemented:

- Runtime stores the just-executed LLM queue item when it opens an effect interaction.
- `pick_interaction_items()` and `score_interaction_target()` prepend this pending item to the active queue.
- `has_llm_plan_for_turn()` remains true while a pending interaction item exists.
- Pending interaction intent is cleared after the effect interaction resolves, after replan/new turn/reset, or after failed ownership.

Reason:

The LLM can choose the correct trainer or ability, but the actual search/discard/target choice often occurs in a later pending-choice step after the original action card has left hand. Without pending ownership, the bridge could no longer see the LLM's exact intent and the rules scorer would silently choose the interaction. That made prompts appear wrong when the bug was actually between route execution and effect interaction.

### 5.12 Route-Critical Energy Preservation

Implemented:

- Raging Bolt LLM blocks Radiant Greninja discard lines when the only discardable energy is a unique missing `Lightning` or `Fighting` requirement for Raging Bolt's primary attack.
- The guard allows the discard when Sada can convert the discard into the missing attack energy on an Ancient attacker.
- Raging Bolt LLM blocks Iono/Research-style hand-reset lines when they shuffle away route-critical missing core energy.

Reason:

The LLM was sometimes using draw engines after the hand already contained the exact energy needed to attack. This is not just a prompt issue: execution must prevent a generic draw action from destroying the known primary route. Raging Bolt still needs draw engines, but not when the action discards or shuffles the only visible `Lightning`/`Fighting` needed to unlock `极雷轰`.

### 5.13 Gust-KO Hard Route Preference

Implemented:

- Runtime route repair now lets `route:gust_ko` override a plain current attack when the candidate route exposes a KO route.
- A generic "current high-pressure attack" no longer blocks repair to a higher-priority gust-KO route.

Reason:

When Boss/Catcher can take a prize or close the game, attacking the active Pokemon is often legal but inferior. This is one of the places where a route candidate is more reliable than asking the LLM to infer prize math from raw board JSON. The runtime should still not hard-ban dead gust, but it should hard-prefer a proven gust-KO candidate route over a weaker active attack.

### 5.14 Empty-Bench Survival Before Attack

Implemented:

- Prompt facts now include `own_bench_count`.
- Candidate attack routes can prepend a `bench_basic` action before terminal attack when own bench is empty.
- The Raging Bolt support-ex bench guard now allows two-prize support basics on an empty bench, but still blocks them as low-priority liability once a stable bench exists.

Reason:

A correct attack can still lose the game if the active Pokemon is knocked out and there are no benched Pokemon. The route layer must treat "bench one Pokemon before attacking" as a survival prerequisite when the bench is empty. This is deck-independent and should become part of the generic LLM strong-mode template.

### 5.15 Burst Attack Interaction Bridge

Implemented:

- The interaction bridge can choose enough board basic energy for Raging Bolt's burst attack to KO the opponent active, respecting max selection.
- Chinese and English energy aliases are normalized for interaction matching.

Reason:

The LLM can decide to use `极雷轰`, but the number and identity of energies discarded are execution-layer details. If the bridge discards too little energy, the route intent is lost even though the attack action was selected correctly.

## 6. Rejected Experiment

Hard runtime blocking of dead Boss/Catcher/gust actions was tested and rejected in this round.

Result:

- Before hard block: 3-game anchor after deckout guard was 2 wins / 3 games against rules Miraidon.
- With hard dead-gust block: 1 win / 3 games against rules Miraidon.

Decision:

Do not keep a hard runtime ban. Keep dead gust as tactical fact/prompt/route guidance first. Only add a hard block if a larger benchmark proves it helps and focused tests define exact allowed exceptions.

For every completed game:

### 7.4 Parallel Validation

Use process-level parallelism, not same-process game concurrency.

Reason:

- `PlayerState` forced shuffle seed is process-global.
- `CardInstance.reset_id_counter()` is process-global.
- recorder paths and LLM audit logs are process-level resources.
- LLM requests are slow enough that serial validation hides iteration speed and long-game stalls.

## 8. Next Iteration Plan

### Step 1: Run 10-Game Rules Miraidon Batch

The 3-game anchor now passes cleanly. The next validation step is a 10-game rules Miraidon batch using the same strong-mode settings and audit output. This should run through `run_llm_duel_parallel.ps1` with `Concurrency=3`, not as one serial Godot process.

Required evidence:

- LLM request success rate remains near 100%.
- No action cap.
- No repeated JSON/timeout fallback.
- No repeated no-bench loss after a successful attack.
- No single game exceeds the configured wall-clock cap without producing a classified failure row.
- Losses, if any, are classified by exact layer: prompt omission, fact bug, missing candidate route, compiler bug, interaction bridge bug, runtime ownership bug, or fallback scorer drift.

### Step 2: Inspect Remaining Losses

Likely remaining failure points:

- Active Raging Bolt has only one required energy and the route layer does not expose a direct Lightning/Fighting completion line.
- Active non-attacker can pivot to Raging Bolt, but candidate route is missing the exact attach/pivot/attack sequence.
- LLM uses gust without a KO route because the fact layer marks it as legal but not clearly resource-negative.
- LLM over-draws or over-filters after the attack route is already available.
- The LLM attacks correctly but fails to maintain a second attacker or survival bench after the first KO exchange.
- The interaction bridge executes a legal but low-damage burst discard instead of the KO-sized discard.

### Step 3: Add Focused Tests From Exact Payloads

Each new fix must get a small regression:

- Bad payload produces the missing candidate route.
- The candidate route materializes into exact legal ids.
- The queue executes in order.
- Interaction bridge picks the intended card/energy/target.
- Replan sees updated hand and turn flags.

### Step 4: Scale Only After Takeover Is Clean

Run 10 games only after the 3-game anchor has:

- no request reliability issue,
- no obvious queue ownership bug,
- no repeated action cap,
- no obvious prompt payload omission.

### Step 5: Promote Template Lessons

Only promote an idea to the generic LLM architecture when it is deck-independent:

- route candidates over free-form action chains,
- low-deck draw/churn lock,
- resource conflict enforcement,
- interaction schema enforcement,
- route-critical discard preservation,
- runtime ownership audit.

Keep deck-specific semantics in deck hooks:

- primary attacker identity,
- low-value attack identity,
- safe setup engines,
- energy assignment priorities,
- deck prompt from editor strategy text.

## 9. Definition Of Done

This architecture round is complete when:

- Focused LLM tests pass.
- Script load tests pass.
- 3-game anchor completes with high takeover and no action cap.
- 10-game rules Miraidon batch completes with high takeover and no repeated architecture-class failure.
- At least one loss has been traced to a concrete layer and converted into a focused test or documented rejected experiment.
- A later 30-game batch shows stable behavior without deckout, repeated low-value first attack, no-bench attack loss, or queue ownership failure.
