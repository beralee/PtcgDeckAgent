# 击倒后手动拿取奖赏卡 UI 设计

## 目标

当前 KO 结算会由引擎直接把奖赏卡加入手牌，玩家没有交互过程。这个方案要改成：

- 击倒对方宝可梦后，由当前玩家自己选择奖赏卡
- 选择入口直接使用左侧现有的 `3x2` 奖赏卡区
- 奖赏卡保持固定槽位，不因为拿走中间某一张而塌缩重排
- 选中后播放翻牌动画，再把卡加入手牌
- 多奖赏 KO 时逐张选择，直到拿完为止

## 现状问题

### 1. 引擎直接自动拿牌

`GameStateMachine._finalize_knockout()` 当前直接从 `player.prizes` 里 `pop_back()` 并加入手牌。  
这会导致：

- UI 无法插入“玩家选择奖赏卡”的交互
- 也没有任何翻牌动画时机

### 2. 奖赏卡数据只有“剩余数组”，没有固定槽位

`PlayerState.prizes` 当前只是剩余奖赏卡数组。  
如果直接 `pop_at()` 或 `erase()`，中间位置会塌缩。UI 即使做成 `3x2` 网格，也无法知道“第 2 格原来是哪张奖赏卡”。

## 方案

### 1. 为奖赏卡增加固定槽位布局

在 `PlayerState` 中新增 `prize_layout`，用于记录奖赏区 6 个固定位置。

- `prizes`：仍然表示“还没拿走的奖赏卡”
- `prize_layout`：表示“奖赏区固定 6 格当前各放了什么”
- 被拿走的格子记为 `null`

这样可以同时满足：

- 规则判断继续使用 `prizes.size()`
- UI 按 `prize_layout` 刷新固定 6 格

### 2. KO 后改为发起选择请求

`GameStateMachine._finalize_knockout()` 不再自动拿牌，而是：

- 先把被击倒宝可梦正常移出场地
- 记录“当前由谁拿奖赏卡、还剩几张要拿、是哪一方被击倒、是否是战斗位 KO”
- 发出 `player_choice_required("take_prize", ...)`
- 暂停 KO 后续流程

### 3. UI 直接在左侧奖赏区完成选择

`BattleScene` 接到 `take_prize` 后：

- 高亮当前玩家自己的奖赏卡区
- 做一个轻量“弹出 / 聚焦”动画
- 允许左键点击有牌的奖赏格

玩家点击后：

- 先在被点中的那一格上播放翻牌动画
- 再调用 `GameStateMachine.resolve_take_prize(player, slot_index)`

### 4. KO 结算恢复顺序

每拿完 1 张奖赏卡：

- 引擎更新手牌和奖赏区
- 如果还有剩余奖赏要拿，继续发下一次 `take_prize`

全部拿完后：

- 先检查是否因拿完奖赏卡直接获胜
- 若击倒的是战斗宝可梦，再进入替换宝可梦流程
- 若击倒的是备战宝可梦，则回到 `POKEMON_CHECK` 继续剩余 KO / 回合推进

## 非目标

- 本次不改奖赏卡查看器右键预览的整体行为
- 本次不引入手牌飞行路径动画，只做奖赏卡翻牌动画
- 本次不改奖赏卡区域现有 `3x2` 的视觉结构

## 测试策略

### 引擎

- 击倒后不再自动把奖赏卡加入手牌
- `take_prize` 请求会按张数逐次触发
- 选择中间槽位后，固定槽位映射不塌缩
- 多奖赏 KO 拿完最后一张后，能正确恢复到替换宝可梦或下一步流程

### UI

- 奖赏区刷新使用固定槽位而不是 `remaining_count`
- 空槽位保留位置，只隐藏卡面
- 奖赏选择激活时，仅当前玩家自己的奖赏区可点击
