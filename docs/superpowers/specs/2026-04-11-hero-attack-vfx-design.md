# 精品英雄池攻击特效设计

日期：2026-04-11

## 背景

当前对战表现层已经有基础的抽牌、弃牌、洗牌等 UI 动画，但攻击演出仍然偏弱，缺少“这只宝可梦打起来就是不一样”的辨识度。

目标不是为所有宝可梦一次性做完整专属特效，而是先围绕游戏内置牌组的主攻手，建立一套可扩展的攻击特效系统，并在第一期交付 20 只主攻手的精品演出。

## 目标

第一期目标：

- 覆盖内置牌组里的 20 只主攻手
- 每只主攻手至少有一套可感知的 signature 攻击演出
- 整体视觉风格折中：清晰、克制，但有记忆点
- 底层采用混合型生产方案
  - 程序化负责时序、命中、节奏、镜头反馈
  - AI 生成少量透明贴图、flipbook、hit decal 等视觉资源
- 不让特效系统深度耦合到每张卡的效果脚本

非目标：

- 第一版不覆盖全部宝可梦
- 第一版不追求 3D 电影级镜头
- 第一版不把特效逻辑塞进每张卡的 effect 文件
- 第一版不实现复杂的可中断、可取消、可回放逐帧编辑器

## 成功标准

- 玩家能明显感知到 20 只主攻手拥有不同“气质”的攻击表现
- 没有配置专属特效的攻击仍能优雅回退到通用命中效果
- 新增一只主攻手时，不需要复制整套动画代码，只需要补 profile 和少量资源
- 演出不会阻塞现有攻击结算、AI 对战、回放、测试流程

## 总体方案

采用三层结构：

1. 通用攻击骨架
2. 攻击模板库
3. 主攻手签名层

### 1. 通用攻击骨架

统一负责所有攻击共有的时间结构：

- 起手
- 蓄力
- 发射/突进
- 命中
- 受击
- 收尾

这一层只处理稳定、通用的演出能力：

- 屏幕轻微震动
- 目标受击闪白/位移
- 通用 hit spark
- 伤害数字出现节奏
- 多段攻击的分段节奏
- 演出节点创建和回收

这层不关心“喷火龙”和“多龙巴鲁托”的个性区别，只负责演出框架和运行时稳定性。

### 2. 攻击模板库

第一期不按“每只宝可梦一个独立系统”开发，而是先做 6 到 8 个高质量模板簇：

- 直线喷射
- 投射物连发
- 近身斩击
- 区域爆裂
- 暗影侵蚀
- 雷落点打击
- 龙息冲击
- 水流/波动推进

每个模板定义：

- 攻击起始位置
- 目标路径
- 主视觉层数量
- 命中时的分层反馈
- 适配单体 / bench / 多段攻击的参数
- 可替换资源槽位

模板只做“打法形态”，不做具体宝可梦人格。

### 3. 主攻手签名层

第一期的精品感，来自签名层。

每个主攻手在模板之上增加一层轻量 signature：

- 专属色彩
- 特定尾迹
- 标志性命中 decal
- 特殊蓄力图案
- 特定节奏或镜头反馈

例子：

- 喷火龙ex：烈焰聚爆 + 灼烧爪痕
- 多龙巴鲁托ex：幽灵尾迹 + 幻影分裂命中
- 猛雷鼓ex：重击震地 + 裂隙电弧
- 雷吉铎拉戈VSTAR：龙息聚束 + 龙系残影

签名层只改“味道”，不推翻模板。

## 资源生产策略

第一期采用混合型资源路线。

### 程序化部分

用 Godot 现有运行时体系控制：

- Tween
- 节点动画
- 轻量 shader
- 粒子或粒子等价实现
- 屏幕震动
- 目标受击位移

程序化层负责：

- 演出时序
- 命中位置
- 多段节奏
- 资源拼装
- 回退逻辑

### AI 资源部分

AI 不负责“决定时序”，只负责生产视觉素材：

- 透明 PNG
- flipbook 序列
- hit decal
- aura / streak / burst 图层

适合 AI 生成的资源包括：

- 火焰束
- 斩击纹
- 暗影团
- 雷击枝杈
- 龙息能量纹
- 水流冲击纹
- 超能波纹

不让 AI 直接产出整套最终特效逻辑，否则后续维护和统一风格会失控。

## 数据驱动结构

### AttackVfxRegistry

职责：

- 从卡牌或攻击标识映射到一个 `attack_vfx_profile`
- 为未配置的攻击提供通用 fallback

输入：

- 卡牌标识
- 攻击名或攻击索引

输出：

- `AttackVfxProfile`

### AttackVfxProfile

一个 profile 至少包含：

- `template_id`
- `signature_id`
- `primary_color`
- `secondary_color`
- `projectile_asset`
- `impact_asset`
- `trail_asset`
- `charge_asset`
- `camera_shake_level`
- `hit_flash_level`
- `timing_preset`
- `supports_multi_hit`
- `supports_bench_target`

### AttackVfxPlayer

职责：

- 接收攻击事件
- 读取 profile
- 按模板和 signature 播放完整演出
- 在结束后回收节点

它需要是一个独立播放器，而不是每张卡牌自己控制。

## 第一期开工范围

第一期只覆盖：

- 游戏内置牌组中的 20 只主攻手

选择原则：

- 主打输出角色优先
- 对局出场频率高优先
- 玩家一眼能看出差异的卡优先

第一期交付内容：

- 20 个主攻手 profile
- 6 到 8 个模板簇
- 一套 fallback 通用命中演出
- 第一批 AI 生成 signature 资源包

## 运行时数据流

1. 攻击被宣布或进入结算
2. 战斗层抛出统一攻击演出事件
3. `AttackVfxRegistry` 查找 profile
4. `AttackVfxPlayer` 根据模板和资源开始播放
5. 命中时同步受击反馈
6. 结算完成后清理临时节点

关键约束：

- 特效触发要依赖稳定的攻击事件，而不是散落在每张 effect 脚本内部
- 回放、AI 对战、低配或关闭特效模式必须能平滑降级

## 与现有系统的关系

需要尽量和现有 BattleScene 重构方向解耦。

推荐边界：

- 战斗逻辑层只发攻击演出事件
- VFX 系统自己消费事件并执行表现
- 不修改具体卡牌效果文件去硬编码特效节点

这样后续 BattleScene 大重构时，攻击特效系统不会和 effect 实现绑死。

## 错误处理与降级

如果 profile 不存在：

- 回退到通用命中效果

如果 signature 资源缺失：

- 保留模板
- 仅跳过 signature 图层

如果目标是 bench：

- 使用 bench 目标定位规则

如果是多段攻击：

- 模板按 `timing_preset` 分段播放

如果运行在回放 / AI / 低性能模式：

- 可关闭部分高成本图层
- 保留最小必要命中反馈

## 测试策略

重点不是测试“好不好看”，而是测试系统是否稳。

需要覆盖：

- 20 只主攻手都能正确命中 registry
- 没配置 profile 的攻击能回退
- 单体攻击目标定位正确
- bench 伤害目标定位正确
- 多段攻击时序不会错位
- 演出结束后节点会被正确回收
- 演出不会阻塞攻击结算或 UI 状态推进

建议增加两类测试：

- 数据层测试：profile / registry / fallback
- 运行时测试：player 节点创建、播放、清理

## 实施顺序

1. 列出内置牌组的 20 个主攻手
2. 给 20 只主攻手做模板归类
3. 搭 `AttackVfxRegistry`
4. 搭 `AttackVfxPlayer`
5. 实现 6 到 8 个模板簇
6. 补 20 个 profile
7. 批量生产 AI 资源
8. 逐只微调 signature
9. 加回退与测试

## 风险

### 风险 1：过早逐只手工化

如果直接按“每只独立演出”开工，第一期后续必然无法扩展。

控制方式：

- 先锁模板簇，再做 signature

### 风险 2：AI 资源风格漂移

如果每次生成 prompt 不稳定，视觉会散。

控制方式：

- 固定一套资源生成风格规范
- 统一尺寸、透明边界、色彩约束

### 风险 3：BattleScene 耦合过深

如果直接把演出逻辑塞进战斗主场景，后续重构成本会高。

控制方式：

- 保持 registry / player / profile 独立

## 推荐结论

第一期采用：

- 精品优先
- 20 主攻手
- 混合型资源路线
- 6 到 8 个模板簇
- 每只一层 signature
- 数据驱动的 registry/profile/player 架构

这是当前质量、扩展性、实现成本三者之间最平衡的方案。
