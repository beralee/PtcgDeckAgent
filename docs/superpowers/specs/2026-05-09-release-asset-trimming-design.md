# 发布包资源瘦身设计

## 目标

在不改变现有功能、玩法、UI 入口、资源引用路径和用户操作体验的前提下，先对发布包中的大体积资源做等价瘦身：

- 卡图 WebP 化
- BGM 重新编码
- 中文字体裁剪
- 大图压缩

本轮只处理资源本体。除非验证发现现有加载链路不能兼容瘦身后的资源，否则不改战斗、卡组、AI、更新、云函数或 UI 逻辑。

## 当前体积判断

最近一次发布包中，主要体积来源不是代码，而是内置资源：

- `data/bundled_user/cards/images/**/*.png.bin`：约 50 MB
- `assets/audio/bgm/*.mp3`：约 26 MB
- `assets/fonts/NotoSansSC-VF.ttf`：源文件约 17 MB，导出后约 12.7 MB
- `assets` 下 UI 背景、标题图、演示图和 VFX 纹理：导出后约 31 MB

因此本轮优化优先级是：卡图、音乐、字体、大图。引擎二进制、脚本和 JSON 数据不是第一阶段目标。

## 总体原则

1. 保持资源路径不变

   现有代码和场景引用的路径不改。例如卡图仍通过 `CardData.build_bundled_image_path()` 得到 `res://data/bundled_user/cards/images/{set}/{idx}.png.bin`，BGM 仍保持当前 catalog 路径，字体仍是 `res://assets/fonts/NotoSansSC-VF.ttf`。

2. 保持资源语义不变

   同一张卡仍显示同一张卡图，同一首 BGM 仍是同一首曲子，同一张背景图仍表达同一画面。允许做编码、尺寸和压缩参数优化，不允许替换内容含义。

3. 保持加载链路不变

   卡图读取仍走现有 `FileAccess` 加字节头识别逻辑；BGM 仍走 `BattleMusicManager`；字体仍走 `FontBootstrap`；场景贴图仍走 Godot 资源导入。

4. 保持离线体验不变

   不把卡图、音乐或字体移到云端，不引入首启下载，不引入按需加载资源包。安装后离线仍能正常看卡图、编辑卡组、战斗、播放内置音乐。

5. 原始资源不放进导出路径

   如果需要保留临时原图或原音频，只能放在 `.tmp`、外部工作目录或其它不会被 Godot export 收进包里的位置。不能在 `assets/` 或 `data/` 下面留下 `_original`、`backup` 之类副本。

## 卡图 WebP 化

### 现状

内置卡图路径为：

`res://data/bundled_user/cards/images/{set}/{idx}.png.bin`

启动时 `CardDatabase._seed_bundled_user_data()` 会把 `.bin` 后缀去掉，复制到：

`user://cards/images/{set}/{idx}.png`

卡图显示点目前通过读取文件字节并按文件头识别格式，相关位置已经支持 PNG、JPG、WebP：

- `scenes/battle/BattleCardView.gd`
- `scenes/deck_manager/DeckManager.gd`
- `scenes/deck_editor/DeckEditor.gd`
- `scripts/ui/decks/DeckViewDialog.gd`

### 方案

将 `data/bundled_user/cards/images/**/*.png.bin` 的内部字节从 PNG 转为高质量 WebP，但保留文件名和路径不变。

也就是说：

- 文件路径仍叫 `.png.bin`
- 新安装用户复制到 `user://` 后仍叫 `.png`
- 文件内容实际是 WebP
- 加载器通过文件头识别 WebP，不依赖扩展名

### 质量标准

建议先使用 WebP lossy quality `88-92`，并保留 alpha 支持。卡牌图本身主要在卡组、手牌、详情弹窗中显示，目标是正常游戏距离下无可感知损失。

如果某些卡图文字边缘出现压缩痕迹，则对这些卡单独提高质量参数，或者退回 PNG。

### 风险与防线

- 风险：某个遗漏的显示入口仍按扩展名当 PNG 加载。
- 防线：资源替换前后用 `rg` 检查所有卡图加载点，确认没有直接 `load("*.png")` 读取 `user://cards/images` 的逻辑。
- 风险：老玩家 `user://cards/images` 已有旧 PNG，不会被覆盖。
- 防线：这不影响功能；老玩家继续用旧图，新安装和缺失图片走新 WebP。若以后要统一用户缓存，再单独做版本化迁移，不纳入本轮。

## BGM 重新编码

### 现状

内置战斗音乐位于：

`assets/audio/bgm/*.mp3`

`BattleMusicManager` 支持 `ogg`、`mp3`、`wav`，但当前 catalog 和 fallback 都指向 `.mp3`。

### 方案

第一阶段保持文件名、扩展名和 catalog 路径不变，只把 3 个 MP3 重新编码到更合理的码率。

建议参数：

- MP3
- 128-160 kbps
- 44.1 kHz
- stereo 或 joint stereo
- 保留曲目长度和开头结尾

暂不改成 Ogg，避免为了压缩音频而同步修改 catalog、fallback 路径和用户镜像目录文件名。

### 质量标准

战斗中播放时不应出现明显削高频、破音、音量突变、开头延迟或循环断裂。因为当前用途是背景音乐，目标是听感无明显变化，而不是和源文件 bit-perfect 一致。

### 风险与防线

- 风险：重新编码后循环点变差。
- 防线：逐首进入战斗播放，至少听完开头、高潮段和一次循环衔接。
- 风险：Godot 导入缓存没有刷新。
- 防线：替换源文件后重新打开 Godot 或清理对应 `.godot/imported` 缓存，再重新导出。

## 中文字体裁剪

### 现状

项目通过 `FontBootstrap.gd` 安装：

`res://assets/fonts/NotoSansSC-VF.ttf`

该字体是全量 CJK 字体，体积较大。

### 方案

使用 `fonttools pyftsubset` 生成同路径同文件名的字体子集，保留现有 UI、卡牌文本、卡组文本、推荐文章、设置页、战斗日志中已经使用到的字符。

字符来源应至少扫描：

- `scenes/**/*.gd`
- `scenes/**/*.tscn`
- `scripts/**/*.gd`
- `data/**/*.json`
- `community/**/*.json`
- `docs/cloud_functions` 不需要进字体子集，除非其中字符串会直接显示到客户端

额外保留：

- ASCII
- 全角/半角标点
- 数字、英文、常用数学符号
- 常用中文字符集
- PTCG 常见术语字符：宝可梦、特性、招式、能量、弃牌、备战、进化、奖赏、伤害、抵抗、弱点等

### 质量标准

所有当前客户端可显示文本不能出现 tofu 方框、乱码、缺字或字体明显变化。尤其要覆盖：

- 首页
- 对战设置
- 卡组中心
- 卡组编辑
- 卡牌详情
- 对战 HUD
- 对战结算
- AI 思考文案
- 推荐文章完整解读

### 风险与防线

- 风险：服务端未来推荐文章出现子集外字符。
- 防线：字体子集除了扫描当前文本，还应加入常用中文字符集；如果仍担心未来内容，保留当前全量字体作为回滚方案。
- 风险：Android 对字体子集兼容性不同。
- 防线：必须在 Android 包上检查中文显示，尤其是此前出现过乱码风险的冒号、书名号、右书名号等字符。

## 大图压缩

### 现状

`assets/` 下存在多张首页背景、标题图、演示图、状态图标、VFX 图。它们导出后成为 `.ctex`，占用较大。

### 方案

第一阶段只做同路径、同扩展名压缩，优先保持 PNG 路径不变。

允许操作：

- 去除无用 metadata
- 降低超过显示上限的分辨率
- 调整 PNG 压缩参数
- 对纯不透明大背景，可在确认引用安全后考虑 JPG/WebP，但这属于第二阶段，因为需要改资源路径或重新绑定场景引用

不允许操作：

- 改变图片构图
- 改变 HUD 配色风格
- 删除仍被引用的图片
- 在 `assets/` 下保留原图副本

### 质量标准

主菜单、卡组中心、对战设置、战斗、结算、反馈/关于弹窗和 VFX 动画的视觉效果应与当前版本一致。背景图可以减小到目标设备实际需要的显示尺寸，但不能糊、不能有明显压缩块、不能造成 UI 边缘锯齿。

### 风险与防线

- 风险：误压透明图导致 alpha 边缘脏。
- 防线：透明 UI、VFX、图标必须单独检查 alpha；有脏边的资源回退或提高质量。
- 风险：缩小过度导致 2K/高 DPI 下变糊。
- 防线：首页主背景、标题图按当前最大目标分辨率保留足够像素，移动端和桌面端都截图验收。

## 工具与产物约定

后续实现建议使用一个明确的资源瘦身脚本，避免手工零散处理：

`scripts/tools/optimize_release_assets.ps1`

脚本职责：

- 输出处理前后体积报告
- 只处理白名单资源
- 不处理 `.gd`、`.tscn`、`.json` 逻辑文件
- 不在导出目录留下原始备份
- 失败时停止，不做半套资源替换

建议白名单：

- `data/bundled_user/cards/images/**/*.png.bin`
- `assets/audio/bgm/*.mp3`
- `assets/fonts/NotoSansSC-VF.ttf`
- 经引用审计确认的大图资源

## 验收标准

资源瘦身后必须完成以下检查：

1. 体积检查

   重新导出 Windows、Android、macOS，记录每个平台包体积和各类资源体积变化。

2. 卡图检查

   在卡组中心、卡组编辑、对战手牌、对战场上、卡牌详情弹窗中查看不同类型卡牌：宝可梦、物品、支援者、道具、竞技场、基础能量、特殊能量。

3. 音乐检查

   在战斗中依次播放所有内置 BGM，确认可以播放、暂停、停止、切换和循环。

4. 字体检查

   跑一遍主要页面，重点看中文标点、书名号、冒号、数字、英文模型名、推荐文章和卡牌长文本。

5. 大图检查

   检查首页、设置页、卡组中心、对战页、结算页和特殊状态/VFX 资源，确认没有明显糊图、脏边或拉伸。

6. 回归检查

   至少启动一次本地游戏，进入对战设置，开始一场对战，打开卡牌详情，进入卡组中心，导入/编辑一套卡组。

## 回滚策略

本轮所有改动都应是资源替换，回滚方式应简单：

- 通过 Git 回滚被替换的资源文件
- 删除 `.godot/imported` 中相关导入缓存后重新导出
- 不需要迁移玩家数据
- 不需要改云函数
- 不需要改卡牌效果实现

## 非目标

以下内容不纳入本轮：

- 不做资源包拆分
- 不做首启下载
- 不做按需加载卡图
- 不拆 macOS universal 包
- 不改 Godot 自定义导出模板
- 不改战斗或卡组逻辑
- 不改云函数协议
- 不删除仍被引用的 UI 资源

## 预期结果

在不改变功能和体验的前提下，本轮资源瘦身预计可以显著降低包体：

- 卡图：预计节省 25-35 MB
- BGM：预计节省 14-18 MB
- 字体：预计节省 8-10 MB
- 大图：预计节省 8-15 MB

合计预计节省约 55-78 MB。实际结果以重新导出后的 Windows、Android、macOS 包体积为准。
