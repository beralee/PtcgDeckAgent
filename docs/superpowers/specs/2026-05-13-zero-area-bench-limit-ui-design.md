# 零之大空洞备战区上限与横竖屏布局设计

日期：2026-05-13

## 目标

实现 `CSV9C_207 零之大空洞`：

- 当前竞技场是零之大空洞时，每名玩家独立检查自己场上是否有太晶宝可梦。
- 有太晶宝可梦的一方备战区上限为 8；没有太晶宝可梦的一方仍为 5。
- 竞技场离场，或某一方不再有太晶宝可梦时，该玩家必须把备战宝可梦弃到 5 只。
- 横屏和竖屏 UI 都必须能展示 8 个备战槽，不把其他区域挤出屏幕。

本次不改变卡牌图片、卡组导入格式、回放数据结构和 AI 策略偏好，只修正规则合法性、场地清理和对战场面布局。

## 规则层设计

现有 `BenchLimitHelper.get_bench_limit(state)` 是全局上限，能支持崩塌的竞技场这类双方同限场地，但无法表达零之大空洞的“按玩家分别判定”。改造为：

- 保留 `get_bench_limit(state)`，供旧调用保持兼容。
- 新增 `get_bench_limit_for_player(state, player)`，所有放置备战宝可梦的规则检查都走玩家维度。
- 崩塌的竞技场仍返回 4。
- 零之大空洞在场时：
  - `player` 场上任意战斗区或备战区宝可梦是太晶，则返回 8。
  - 否则返回 5。

太晶判定复用 `CardData.is_tera_pokemon()`，同时保留名称/标签兜底，避免导入卡牌标签不完整时漏判。

## 清理时机

零之大空洞失效清理不是击倒，不拿奖赏，不触发 KO 相关效果；被清理宝可梦及附属卡进入其持有者弃牌区。

需要在这些状态变化后执行清理：

- 使用新竞技场替换零之大空洞。
- 道具、招式或特性把当前竞技场移出场。
- 进化、回手、洗回牌库、弃置、击倒等导致玩家场上不再有太晶宝可梦。

实现上由 `GameStateMachine.enforce_current_bench_limits()` 统一执行，关键动作结束后调用。若双方都需要弃到 5，按当前竞技场持有者开始；如果零之大空洞已经离场，则使用离场前记录的持有者，否则用当前回合玩家兜底。

当前项目没有通用的“场地失效后追加选择交互”管线。为了先保证规则正确，未提供显式选择时按从最右侧备战槽开始弃置；有上下文选择时按选择结果执行。后续可在通用待处理选择队列里补充 UI 选择。

## UI 设计

### 动态槽位

场景启动时最多创建 8 个备战槽，但默认只显示 5 个。显示数量由当前场面决定：

- `max(5, 我方上限, 对方上限, 我方当前备战数, 对方当前备战数)`，上限封顶 8。
- 这样默认布局不被压缩；只有零之大空洞生效或已有超 5 备战时才展示 8 格。

### 横屏

横屏继续使用单排备战区。

- 当显示 8 格时，使用实际槽位数重新计算卡牌高度。
- 缩小备战间距，优先让 8 张卡在中场宽度内横向放下。
- 如果宽度不足，卡牌高度跟随宽度缩小，并允许比默认 5 格布局更低的最小高度。

### 竖屏

竖屏在显示 8 格时使用 4x2 两行备战区。

空间处理顺序：

1. 手牌区按“至少横向容纳 6 张手牌”重新计算手牌卡尺寸，并且展开 8 格时手牌 HUD 高度直接等于手牌卡高度，不再额外保留滚动条和面板 padding。
2. 将手牌 HUD 释放出来的高度回流给场面区，战斗区卡牌和备战区卡牌使用同一个场面缩放因子按比例调整。
3. 场上卡牌的 HP、状态、能量、道具等覆盖层只跟随卡牌实际尺寸缩放，不再因为竖屏模式套用固定放大倍率。
4. 两行备战区固定 4 列，不通过挤压其他区域解决空间问题。

默认 5 格时仍保持单行，避免不需要零之大空洞时改变用户体验。

## 实施计划

1. 完善 `CSV9C207AreaZeroUnderdepths.gd`，提供 effect_id、玩家太晶判定、玩家维度上限、清理执行和可测试 API。
2. 注册 effect_id，并让 `BenchLimitHelper` 支持玩家维度上限。
3. 在 `GameStateMachine` 中集中调用备战上限清理，覆盖场地替换、训练家/场地效果、进化、击倒和回合中效果执行后。
4. UI 创建 8 个槽位，布局层按当前显示数量计算横屏/竖屏尺寸。
5. 补充规则测试、布局测试、竖屏命中测试和注册测试。
6. 跑定向测试、脚本加载回归和 `$card-audit`。
