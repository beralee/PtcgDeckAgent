# 非战斗竖屏布局系统设计

## 状态

设计稿。本文只做系统分析和后续实现方案，不修改对战模式、不修改卡组编辑器，也不改变任何卡牌或战斗逻辑。

## 背景与目标

当前首页右下角已经能看到横竖屏切换按钮，但点击后没有明显反应。原因不是按钮本身没有改变状态，而是非战斗页面只有“屏幕方向偏好”和系统方向调用，没有像战斗模式那样在页面层执行重新测量、重新排版和控件尺寸调整。

本次目标是为以下非战斗页面设计一套完整的竖屏布局系统：

- 首页
- 对战设置
- 卡组中心首页
- 比赛模式整体：选参赛卡组、比赛设置、比赛总览、最终排名
- 对战复盘首页
- AI 设置

竖屏模式必须面向手机用户重新排版：单列、可滚动、大按钮、大字号、底部主操作、避免横向挤压和误触。

## 非目标

- 不改 `scenes/battle`、`scripts/ui/battle` 的战斗布局代码。
- 不改战斗中的 `battle_layout_mode`、战斗 HUD、行动 HUD、奖赏卡、撤退、卡牌交互。
- 不改卡组编辑器页面和卡组编辑器内部逻辑。
- 不把非战斗横竖屏偏好和战斗横竖屏偏好合并。
- 卡组编辑页面因为交互密度和卡牌网格复杂度过高，保持永久横屏；本设计只允许在进入卡组编辑器前切回横屏方向，不为编辑器做竖屏重排。

## 当前问题分析

当前非战斗切换链路大致是：

1. 首页按钮调用 `GameManager.toggle_non_battle_layout_mode()`。
2. `GameManager.non_battle_layout_mode` 在 `landscape` 和 `portrait` 间切换。
3. 移动端调用 `DisplayServer.screen_set_orientation()`。
4. 其他非战斗页面进入前调用 `apply_non_battle_orientation()`。

这个链路缺少关键的一步：页面没有收到“现在要变成竖屏布局”的统一上下文，也没有执行页面级布局切换。

因此在桌面或部分 Android 环境中会表现为：

- 按钮图标或持久化状态可能变了，但当前页面没有重新布局；
- 页面仍保持原来的固定宽度面板、横向按钮行或双栏结构；
- 系统方向变化和 Godot resize 是异步的，即使稍后 viewport 尺寸变了，也没有页面级 `_apply_non_battle_layout()` 去响应；
- 对战设置、比赛总览、AI 设置等页面本来就是横向面板设计，竖屏下只靠系统旋转无法得到可用 UI。

## 战斗模式可参考的设计

战斗模式不是只调用屏幕方向 API，而是有完整的布局决策和应用层：

- `BattleLayoutController.resolve_layout_mode()` 根据偏好、viewport、移动端状态决定实际布局。
- `BattleLayoutCoordinator.apply()` 生成布局上下文，切换横屏 view 或竖屏 view。
- `BattlePortraitLayoutView.apply()` 在竖屏下重新计算安全宽度、卡牌尺寸、HUD 位置、手牌区和场地区。
- 战斗场景在 ready、resize、状态变化后持续重新应用布局，并做 deferred finalization。

非战斗系统应该复制这套思想，但不要复用战斗类本身。战斗布局包含大量卡牌区、HUD 和画布旋转规则，直接复用会把两套系统耦合起来，后续更难维护。

## 推荐架构

新增一套非战斗专用布局基础设施，命名可以保持明确边界：

- `scripts/ui/non_battle/NonBattleLayoutController.gd`
- `scripts/ui/non_battle/NonBattleLayoutContext.gd` 或 Dictionary context
- `scripts/ui/non_battle/NonBattleResponsivePage.gd` 或轻量 helper

### 状态来源

继续保留 `GameManager.non_battle_layout_mode`，可选值先保持：

- `landscape`
- `portrait`

第一版不新增用户可见的 `auto` 档。用户看到的仍然是 `横屏` / `竖屏` 两档，但系统要在首次运行或没有明确用户偏好时，根据设备环境主动选择更合理的初始值。

`GameManager.battle_layout_mode` 保持不变，只服务战斗。

### 手机与浏览器默认策略

默认策略需要解决手机用户第一次打开就难以阅读和操作的问题：

- 如果检测到 Android 原生、iPhone/iPad 原生、移动浏览器，且当前没有用户保存过非战斗布局偏好，则首页主动初始化为 `portrait`；
- 如果是浏览器运行，并且 viewport 为竖屏或窄屏触控环境，也按手机处理，首页主动初始化为 `portrait`；
- 如果是桌面浏览器宽屏、Windows/macOS 桌面运行，默认仍为 `landscape`；
- 如果用户已经手动点过首页横竖屏按钮，则尊重用户保存的 `non_battle_layout_mode`，不要每次启动都覆盖；
- 横竖屏按钮仍保留，手机用户可以主动切回横屏；
- 切换后要立刻应用当前页面布局，并持久化用户选择。

手机/浏览器检测建议分层判断：

1. `OS.get_name()` 为 `Android` 或 `iOS` 时直接视为手机/平板移动端。
2. Web 导出环境中，如果能拿到 user agent，则 Android、iPhone、iPad、Mobile Safari、移动 Chrome 视为移动浏览器。
3. Web 导出环境中，如果 user agent 不可用，则用 viewport 和触控能力兜底：竖屏、宽度低于手机阈值、或触控指针环境命中任一条件时视为移动浏览器。
4. Headless 测试环境只使用显式传入的 viewport 和平台参数，不读取真实系统 API。

对战设置里的“战斗横竖屏选项”也需要移动端默认值：

- 如果没有保存过 battle setup 偏好，并且当前环境是手机或移动浏览器，则默认选择战斗竖屏；
- 如果已有保存的 `battle_layout_mode`，尊重保存值；
- 这个默认只改变对战设置页的初始选择，不修改战斗布局系统实现；
- 文案上要明确这是“对战中布局”，不要和首页右下角的“菜单横竖屏”混淆。

卡组编辑器的方向策略：

- 从任何非战斗页面进入卡组编辑器前，强制应用横屏方向；
- 卡组编辑器不监听 `non_battle_layout_mode_changed`；
- 从卡组编辑器返回卡组中心首页时，再恢复非战斗页面自己的横竖屏偏好；
- 不添加卡组编辑器竖屏入口、不做竖屏提示流、不改编辑器内部控件。

### 信号与应用流程

`GameManager.set_non_battle_layout_mode()` 后除了设置系统方向，还应该发出非战斗布局变化信号：

```text
non_battle_layout_mode_changed(mode)
```

每个目标非战斗页面在 `_ready()` 中注册：

```text
_apply_non_battle_layout()
```

并在以下时机重排：

- 页面 ready 后；
- `NOTIFICATION_RESIZED`；
- 收到 `non_battle_layout_mode_changed`；
- 移动端切换方向后的 1-2 帧 deferred pass，兜底 Android 异步 resize。

### 布局上下文

非战斗 controller 输出统一 context：

```text
{
  viewport_size,
  resolved_mode,
  is_portrait,
  safe_rect,
  content_width,
  page_margin,
  section_gap,
  title_font_size,
  body_font_size,
  meta_font_size,
  button_font_size,
  primary_button_height,
  secondary_button_height,
  input_height
}
```

第一版建议竖屏基础指标：

- 页面左右安全边距：20-28 逻辑像素，窄屏取小值；
- 标题字号：30-36；
- 分区标题字号：21-24；
- 正文字号：17-19；
- 辅助信息字号：14-16；
- 主按钮高度：64-72；
- 次按钮高度：56-64；
- 输入框高度：52-60；
- 列表项最小高度：88-104；
- 页面主体全部单列滚动，除卡图/卡组横向预览外不允许横向滚动。

## 页面级设计

### 首页

当前首页是居中菜单加右下角圆形按钮组。竖屏下应改为手机首页：

- 背景仍铺满；
- Android、iPhone、移动浏览器首次打开且没有手动偏好时，首页直接进入竖屏排版，不需要玩家先点切换按钮；
- 主菜单改为单列，宽度使用安全区宽度的 82%-90%，最大不超过 420；
- 主按钮高度提高到 64-72，字号提高到 20-22；
- 按钮顺序保持现有业务优先级：开始对战、卡组中心、比赛模式、对战复盘、AI 设置、退出；
- 右下角工具按钮保留，但在竖屏下改成底部或右侧竖向浮动工具条，避免遮挡主按钮；
- 横竖屏切换按钮点击后必须立即重排当前首页，不能只等下一次进场景；
- 含羞苞动画区域限定在菜单外侧的底部活动区，不能覆盖按钮，也不能拦截按钮输入；
- 版本号放在底部安全区内，字号 13-15，优先不抢主操作注意力。

首页按钮“点了没反应”的验收标准是：点击切换按钮后，当前页面在 1 秒内明显变为另一套排版，按钮尺寸和菜单位置发生可见变化。

### 对战设置

当前对战设置是左右双栏：左侧基础设置，右侧 AI/BGM/开始按钮。竖屏下改为单列表单：

- 顶部为标题和返回入口；
- 手机/移动浏览器首次进入时，战斗布局选项默认选中竖屏；已有用户保存值时不覆盖；
- 主体放入 `ScrollContainer`；
- 分区顺序：
  1. 模式选择；
  2. 玩家 1 卡组；
  3. 玩家 2 / AI 卡组；
  4. 先后手；
  5. 场地背景和动态场馆；
  6. 战斗横竖屏设置；
  7. AI 强度、BGM 等高级设置；
- 每个卡组选择区域使用全宽大按钮，查看/编辑按钮在下方横向并列或换行；
- 现有“战斗横竖屏设置”只控制进入战斗后的布局，文案和状态要避免和“非战斗横竖屏”混淆；
- 开始对战和返回按钮放底部 sticky action bar，开始对战为主按钮，高度 68 左右；
- 不改变现有战斗设置保存结构和战斗偏好语义。

### 卡组中心首页

不触碰卡组编辑器，只改卡组中心列表首页。

竖屏设计：

- 顶部为 app bar：标题、返回；
- 导入卡组、同步卡图改为标题下方全宽操作区，按钮高度 58-64；
- 推荐卡组区域变为单列卡片，按钮区允许换行，不再依赖横向一排塞满；
- 本地卡组列表每项改为竖向卡片：
  - 第一行：卡组名和基础信息；
  - 第二行：查看、编辑、重命名、删除按钮，窄屏下 2x2 网格；
- 空状态文案居中，按钮引导改为页面内导入按钮，不再说“右上角”；
- 导入弹窗改为底部 sheet 或全宽居中 modal，宽度使用安全区，输入框高度至少 56，导入/关闭按钮高度至少 58。
- 点击“编辑”进入卡组编辑器前必须切换为横屏；返回卡组中心首页后恢复当前非战斗布局偏好。

### 比赛模式整体

比赛模式由多页组成，应统一成同一套手机赛事流程。

#### 参赛卡组选择

- 固定宽度 720 面板改为全屏单列；
- 当前参赛卡组作为大卡片展示；
- “选择卡组”按钮高度 72，支持两行文本但不溢出；
- 卡组 picker 弹窗在竖屏下使用全屏或底部 sheet；
- picker 内搜索框高度 56，分类 tab 横向滚动或 2 列换行；
- 卡组选项从网格挤压改为单列卡片，点击整张卡片选择；
- 下一步为底部主按钮，返回为次按钮。

#### 比赛设置

- 固定宽度 660 面板改为单列表单；
- 参赛人数、轮数等选项使用全宽大控件；
- 当前卡组摘要放在表单顶部；
- 开始比赛固定在底部主按钮区。

#### 比赛总览

当前总览有 `MiddleRow` 横向展示卡组分布和选手名单。竖屏下改为：

- 比赛摘要卡片在顶部；
- 卡组分布和选手名单上下排列；
- 长文本区域使用可滚动卡片，不使用横向双栏；
- “进入第一轮/下一轮”作为底部主按钮；
- “返回修改设置”作为底部次按钮。

#### 最终排名

- 冠军 banner 保持顶部突出，但高度按屏宽压缩；
- 排名表不要用宽表格，改为排名卡片：
  - 名次、选手名、卡组；
  - 胜/负/平、积分、对手分；
- 按钮底部固定，主按钮用于返回首页或再开一赛，次按钮用于返回总览。

### 对战复盘首页

当前复盘列表行是横向信息 + 两个按钮。竖屏下：

- 顶部 app bar：标题和返回；
- 列表卡片单列滚动；
- 每条复盘第一行显示对局标题，第二行显示时间、双方、回合等 meta，允许换行；
- “回放”作为大主按钮，可以全宽；
- “删除”作为次要危险按钮，放在主按钮下方或右侧小按钮；
- 空状态要说明当前没有复盘，并提供返回按钮，不依赖顶部小按钮。

### AI 设置

当前 AI 设置是固定中心表单，运行时又动态生成左右双栏：表单 + ZenMux 引导。竖屏下：

- 全屏单列 `ScrollContainer`；
- 表单放前面，引导放后面；
- ZenMux 配置步骤和常见问题可以做成折叠区，默认只展开配置步骤；
- API 地址、Key、模型、超时、性格输入全部全宽；
- 输入框高度至少 56，字号 17-18；
- 保存、测试连接、返回放底部 sticky action bar；窄屏下保存/测试可以上下堆叠，返回放次级位置；
- 测试状态文案显示在按钮上方，避免弹跳挤压输入区。

## 实现顺序建议

后续实现时建议按以下顺序推进，避免再次把状态堆在单个页面里：

1. 给非战斗 layout controller 写纯单元测试：横屏、竖屏、移动端、桌面、极窄屏。
2. 给移动端/浏览器默认策略写测试：手机首次首页默认竖屏、桌面默认横屏、用户保存值不被覆盖。
3. 给 `GameManager.non_battle_layout_mode` 写信号和持久化测试，确认不影响 `battle_layout_mode`。
4. 给对战设置默认值写测试：手机/移动浏览器首次默认战斗竖屏，已有保存值时不覆盖。
5. 首页接入 helper，只实现首页竖屏可见重排，修复“按钮点了没反应”的最小闭环。
6. 对战设置接入，确保其中的战斗横竖屏选项仍只影响战斗。
7. 卡组中心首页接入，但不进入卡组编辑器代码；只在进入编辑器前强制横屏，返回后恢复卡组中心布局。
8. 比赛模式四个页面统一接入。
9. 复盘首页和 AI 设置接入。
10. 最后跑完整 UI 关键场景测试和脚本加载回归。

## TDD 验收计划

需要新增或扩展的测试类型：

- `NonBattleLayoutController` 纯函数测试：
  - `portrait` 偏好在 390x844、900x1600 下解析为竖屏；
  - `landscape` 偏好在 1600x900 下解析为横屏；
  - 输出字号、按钮高度、边距达到手机触控标准。
- `GameManager` 状态测试：
  - 切换非战斗布局会持久化；
  - 切换非战斗布局不改变 `battle_layout_mode`；
  - 进入战斗场景前不强行套非战斗布局。
  - Android/iOS/移动浏览器首次运行且无保存偏好时，`non_battle_layout_mode` 初始化为 `portrait`；
  - 桌面宽屏和桌面浏览器首次运行仍初始化为 `landscape`；
  - 已保存的用户偏好不会被移动端默认策略覆盖。
- 首页测试：
  - 点击横竖屏按钮后，主菜单按钮高度和菜单位置立即变化；
  - 模拟 Android/iPhone/移动浏览器竖屏首屏时，首页不点击按钮也应用竖屏布局；
  - 含羞苞层 `mouse_filter` 不拦截主按钮；
  - 工具按钮在竖屏安全区内。
- 对战设置测试：
  - 竖屏下 `ContentColumns` 不再横向挤压；
  - 开始按钮高度达标；
  - 战斗布局设置仍保存到原 battle setup 偏好。
  - 手机/移动浏览器首次打开且没有保存 battle setup 偏好时，战斗布局默认竖屏；
  - 保存过横屏偏好后再次进入，不被手机默认策略改回竖屏。
- 卡组中心首页测试：
  - 竖屏下 deck row 操作按钮可见且不溢出；
  - 导入弹窗宽度不超过安全区；
  - 不加载或修改卡组编辑器场景。
  - 从竖屏卡组中心进入卡组编辑器前应用横屏方向，返回后恢复卡组中心竖屏布局。
- 比赛模式测试：
  - 四个赛事页面在竖屏下无横向双栏；
  - deck picker 搜索和选择按钮可触达；
  - 排名页长名单可滚动。
- 复盘首页测试：
  - 复盘行在窄屏下按钮不挤出屏幕；
  - 删除按钮不会覆盖回放按钮。
- AI 设置测试：
  - 输入框宽度跟随安全区；
  - Save/Test/Back 在竖屏下触控高度达标；
  - ZenMux 引导不把主表单挤出首屏。

## 关键验收标准

- Android、iPhone、移动浏览器首次打开首页时，在没有用户保存偏好的情况下默认就是竖屏布局。
- Android 首页点击非战斗横竖屏按钮后，当前页面立即发生可见重排。
- 退出再进游戏后，非战斗横竖屏偏好保持。
- 手机/移动浏览器首次进入对战设置时，战斗布局选项默认是竖屏；保存过偏好后尊重保存值。
- 首页、对战设置、卡组中心首页、比赛模式、复盘首页、AI 设置在竖屏下没有文字截断、按钮重叠、横向滚动或小按钮难点问题。
- 竖屏模式下所有主按钮、列表按钮、输入框都达到手机触控尺寸，普通玩家可以单手完成主要流程。
- 进入战斗后仍完全使用现有战斗横竖屏系统。
- 从战斗返回非战斗页面后，恢复非战斗页面自己的布局偏好。
- 卡组编辑器保持永久横屏，不做竖屏改造；从卡组中心进入和返回时方向切换正确。

## 风险与约束

- Android 方向切换后 viewport resize 可能滞后，页面必须在信号后立即应用一次，并在后续 resize/deferred 再应用一次。
- 现有部分页面用固定宽度 `CenterContainer/PanelContainer`，竖屏不能只改字体，需要重建或重排容器。
- 非战斗页面不要引入类似战斗输入层的复杂拦截，普通 `Button.pressed` 应保持 Godot 原生路径，避免重现之前触摸事件问题。
- 卡组中心首页和卡组编辑器要在场景边界上隔离；只允许调整 `DeckManager` 首页列表、推荐、导入弹窗，不进入编辑器布局。
- 对战设置里同时存在“进入战斗后的布局选项”，文案和状态必须和首页非战斗切换区分开。
- 移动浏览器判断可能受 user agent 限制，需要 viewport/touch 兜底，但桌面浏览器不能被误判成手机竖屏。
- 首次默认竖屏和用户手动偏好要有明确优先级：默认策略只负责“种初始值”，不能在每次启动时覆盖玩家后来保存的选择。

## 暂定产品决策

第一版非战斗切换的用户界面仍只做 `横屏` / `竖屏` 两档，不展示 `自动` 模式。但系统内部要有一次性的移动端默认策略：手机和移动浏览器在没有保存偏好时主动进入竖屏，战斗设置里的战斗布局也默认竖屏。用户手动切换后，以用户选择为准。
